## 왜 프로젝트는 항상 늦어질까

프로젝트를 시작할 때는 **"이번엔 꼭 일정을 맞춰보자"** 라고 다짐한다. 하지만 현실은 항상 다르다.

나의 경우, 초기에는 **"대충 이 기능은 3일, 저 기능은 5일"** 정도로 막연하게 계획을 세웠다. 그러다 보니 프로젝트 막바지에 **"아직 50%밖에 안 끝났는데 마감이 일주일 남았다"** 는 상황이 자주 발생했다.

특히 기억에 남는 프로젝트가 있다. **2개월 프로젝트**라고 시작했는데, 막상 뚜껑을 열어보니 **4개월**이 걸렸다. 중간에 **"이 기능은 왜 안만들어요?"** 라는 말도 나올 정도로 팀 전체가 혼란에 빠졌다.

그때 깨달은 것이 **WBS(Work Breakdown Structure, 작업 분해 구조)의 중요성**이었다. WBS는 단순한 할 일 목록이 아니라 **프로젝트를 성공적으로 끝내기 위한 전략적 도구**다.

## WBS를 왜 써야 할까

**WBS의 핵심 목적은 단 하나다: "빨리 끝내기 위해서"**

여기서 "빨리"라는 것은 무작정 서두른다는 뜻이 아니다. **계획된 일정 안에 최대한 효율적으로 진행**한다는 의미다.

WBS가 없으면:
- **어디까지 왔는지 모른다** - 진척도 파악 불가
- **언제 끝날지 모른다** - 일정 예측 불가  
- **뭘 빼야 할지 모른다** - 우선순위 판단 불가
- **누가 뭘 하는지 모른다** - 책임 소재 불분명

## 오퍼레이팅 vs 프로젝트

WBS를 제대로 활용하려면 먼저 **업무의 성격을 구분**해야 한다:

### 프로젝트 (Project)
- **명확한 시작점과 끝점**이 있는 업무
- **특정 목표 달성**을 위한 일회성 작업  
- **WBS로 관리**: 전체 구조를 분해하여 체계적 접근

```markdown
프로젝트 예시:
- 새로운 결제 시스템 구축
- 모바일 앱 신규 개발
- 레거시 시스템 마이그레이션
```

### 오퍼레이팅 (Operating)
- **지속적이고 반복적**인 일상 업무
- **서비스 운영과 유지보수** 중심
- **JIRA로 관리**: 티켓 기반의 이슈 트래킹

```markdown
오퍼레이팅 예시:
- 일일 서버 모니터링
- 사용자 문의 대응
- 정기적인 데이터 백업
- 핫픽스 및 긴급 패치
```

**두 업무를 명확히 구분해야 하는 이유**: 관리 방식과 우선순위 판단 기준이 완전히 다르기 때문이다.

## WBS 작성의 핵심 원칙

### 1. 작업 단위는 최대 5일, 가능하면 2일 이하

```markdown
❌ 잘못된 예시:
- 사용자 관리 시스템 개발: 20일

✅ 올바른 예시:  
- 사용자 등록 API 개발: 2일
- 사용자 조회 API 개발: 1일
- 사용자 수정 API 개발: 2일
- 사용자 삭제 API 개발: 1일
- 사용자 관리 UI 개발: 3일
```

**왜 작은 단위로 나눠야 할까?**
- **정확한 진척도 파악**: 매일 완료된 작업 확인 가능
- **빠른 문제 발견**: 지연 시 즉시 알 수 있음
- **동기 부여**: 자주 완료되는 성취감

### 2. Priority(우선순위) 설정

Priority는 **"예정된 날짜에 맞추지 못했을 때 어떤 기능을 빼고 나갈지"** 를 결정하는 기준이다.

```markdown
Priority 기준:
P1 (High): 핵심 사용자 경험에 필수
P2 (Medium): 있으면 좋지만 없어도 됨
P3 (Low): 나중에 추가해도 되는 기능

예시 - 쇼핑몰 프로젝트:
P1: 회원가입/로그인, 주문 내역
P2: 상품 리뷰, 위시리스트
P3: 추천 시스템, 쿠폰 기능
```

### 3. Original Estimation vs Current Estimation

**Original Estimation**: 처음 계획할 때 설정한 예상 시간 (변경 금지)
**Current Estimation**: 실제 진행하면서 조정되는 예상 시간

```markdown
작업: 결제 API 개발
Original Estimation: 3일
Current Estimation: 5일

→ 2일 초과 원인:
- 외부 결제 모듈 연동 이슈
- 추가 보안 요구사항 발생
```

**왜 Original을 고정해야 할까?**
- **추정 정확도 개선**: 나중에 비교 분석하여 경험치 축적
- **생산성 예측**: 팀의 실제 개발 속도 파악
- **계획 능력 향상**: 무엇을 놓쳤는지 학습

## 실무 WBS 활용법

### 1. 일정 지연 대응

**핫픽스나 운영 업무로 인한 지연 시**:

```markdown
상황: 긴급 보안 패치로 하루 소요

실제 진척도 기록 방식:
- 월요일: 프로젝트 작업 (1일)
- 화요일: 긴급 핫픽스 (1일) 
- 수-금: 프로젝트 작업 (3일)

→ 해당 주 실제 프로젝트 진행: 평일 5일 중 4일만 진행

WBS 관리:
- 마일스톤 날짜는 유지
- 실제 투입 시간만 정확히 기록
- "이번 주는 4일만 작업했음" 형태로 관리
```

### 2. 실제 작업 시간 산정

**나의 하루는 몇 시간인가?**

```markdown
일반적인 착각: 하루 = 8시간 개발

현실적인 계산:
- 전체 업무 시간: 8시간
- 회의, 메일, 브레이크: 2시간
- 실제 개발 시간: 6시간

레귤러 업무가 있는 경우:
- 운영 모니터링: 1시간
- 일일 이슈 대응: 1시간
- 순수 개발 시간: 4시간

→ 프로젝트 일정은 실제 개발 가능 시간 기준으로 계산
```

### 3. 진척도 관리

```markdown
주간 진척도 체크:
- 완료된 작업: 실제 소요 시간 기록
- 진행 중인 작업: 남은 시간 재추정
- 대기 중인 작업: 의존성 이슈 확인

마일스톤별 리뷰:
- Original vs Current 비교 분석
- 지연 원인 분석 및 개선점 도출
- 다음 마일스톤 계획 조정
```

## WBS 작성 실전 가이드

### 1. Top-Down 접근법

```markdown
1단계: 주요 기능별 분해
- 사용자 관리
- 상품 관리  
- 주문 관리
- 결제 처리

2단계: 기능별 세부 작업 분해
사용자 관리 → 
  - 회원가입 기능
  - 로그인 기능
  - 프로필 관리 기능

3단계: 구현 단위로 최종 분해
회원가입 기능 →
  - 회원가입 API 개발 (2일)
  - 이메일 인증 구현 (1일)
  - 회원가입 UI 개발 (2일)
```

### 2. 의존성 관리

```markdown
작업 간 의존성 명시:
- 로그인 API → 프로필 관리 API
- 상품 API → 장바구니 API → 결제 API

병렬 처리 가능한 작업 식별:
- UI 개발 ∥ API 개발 (다른 담당자)
- 프론트엔드 ∥ 백엔드 (API 스펙 합의 후)
```

### 3. 리스크 관리

```markdown
불확실성이 높은 작업:
- 외부 시스템 연동 → 버퍼 시간 추가
- 신기술 적용 → 프로토타입 우선 개발
- 복잡한 비즈니스 로직 → 설계 리뷰 먼저

버퍼 시간 할당:
- 전체 프로젝트의 15-20%
- 마일스톤별로 분산 배치
```

## 마무리

처음 WBS를 작성할 때는 **"이렇게 세세하게 나눠야 하나?"** 라는 생각이 들 수 있다. 나도 그랬다.

하지만 실제로 적용해보니, 프로젝트의 **투명성과 예측 가능성**이 크게 향상되는 것을 경험했다. 더 이상 **"언제 끝나냐"** 는 질문에 막막해하지 않아도 되고, **일정 지연 시에도 체계적으로 대응**할 수 있게 되었다.

가장 좋은 점은 **다른 팀들과의 소통이 훨씬 원활해진다**는 것이다. 프로젝트 진행 상황에 대한 명확한 근거가 있다 보니, 기획팀이나 마케팅팀, 경영진과의 협업에서 **서로를 이해하고 신뢰할 수 있는 기반**이 만들어진다.

실제로 WBS를 제대로 활용한 프로젝트에서는:
- **"지금 몇 퍼센트 완료됐나?"** 라는 질문에 정확한 답변 가능
- 일정 지연 시 **우선순위 기반으로 즉시 조정** 가능  
- 팀원들의 **업무 분배와 진척도 파악**이 명확해짐

물론 처음에는 작업 분해하는 데 시간이 걸린다. 하지만 전체 프로젝트 관점에서 보면 **절대 손해 보지 않는 투자**라고 확신한다.

**한 가지 중요한 점**: WBS는 **한 번 만들고 끝이 아니다**. 프로젝트 진행하면서 지속적으로 업데이트하고, 완료 후에는 **Original vs Current 비교 분석**을 통해 다음 프로젝트의 추정 정확도를 높여야 한다.

특히 팀 단위로 개발할 때는 WBS가 **모든 소통의 기준점**이 된다. **"언제까지 뭘 할 것인가"** 가 명확해지면, 놀랍게도 팀 전체의 집중도와 생산성이 크게 향상된다.

WBS 작성, 처음에는 번거롭지만 **한 번 익숙해지면 없어서는 안 될 프로젝트 관리 도구**가 될 것이다. 다음 프로젝트에서는 꼭 시도해보기를 추천한다.
