앱 전환 중심 구조를 웹 완결형 사용자 경험으로 재구성한 사례

웹 확장 · Web MAU 8만 → 12만 · 전체 MAU 25만+

문제 인식

앱 소개용 링크 페이지로는 웹 유입을 실제 사용 전환으로 연결하기 어려웠습니다. 웹에서도 로그인, 상태관리, 결제까지 이어지는 완결된 경험이 필요했습니다. 기존 웹은 소개 페이지 중심이어서 실제 서비스 화면에서 필요한 회원 상태, 프로필, 인증 토큰 같은 상태를 다루는 구조가 없었습니다.

분석

단순히 랜딩 페이지를 개선하는 것만으로는 전환 문제를 해결하기 어렵다고 판단했습니다. 웹만으로 서비스를 이용하고 싶은 수요가 이미 있었고, 실제 전환을 만들려면 웹도 앱처럼 로그인 이후 탐색과 결제까지 이어지는 완결된 구조를 가져야 했습니다. 또 웹과 앱 URI가 분리된 상태로는 기능을 늘릴수록 유지보수 비용이 커질 수밖에 없어서, 상태관리와 라우팅 체계까지 함께 정리하는 방향으로 접근했습니다.

실행

  • 앱 소개 중심이던 웹을 실제 서비스 진입점으로 바꾸고, 탐색부터 결제까지 이어지는 흐름을 웹 안에서 완결할 수 있도록 재구성했습니다.
  • 로그인 상태와 서비스 데이터를 안정적으로 다룰 수 있는 상태 관리 구조를 정리해 주요 화면 경험을 일관되게 맞췄습니다.
  • 웹에서 구현한 핵심 화면을 앱에서도 재사용할 수 있도록 연결 구조를 정리해 플랫폼별 기능 중복을 줄였습니다.

결과

  • 웹 MAU가 8만에서 12만으로 증가했고, 전체 MAU는 25만+ 규모로 확장됐습니다.
  • 앱과 웹의 사용자 흐름을 일관되게 맞춰 기능 확장 시 혼선을 줄였습니다.
  • 웹에서 구현한 VOD 경험을 앱 상세/플레이어 화면에서도 재사용할 수 있게 만들어 플랫폼별 기능 중복을 줄였습니다.
  • 웹뷰 캐시와 브리지 기반 통신을 통해 앱 안에서도 웹 기능을 자연스럽게 연결할 수 있는 구조를 만들었습니다.

Before

1 웹은 서비스 소개와 앱 다운로드 유도 중심의 랜딩 페이지
2 실제 탐색, 로그인, 결제는 모두 앱에서만 가능
3 웹 유입이 실제 사용 전환으로 이어지기 어려움
4 웹과 앱의 사용자 흐름이 분리되어 이용 경험이 자주 끊김

After

1 서비스 상세 페이지를 실제 서비스 진입점으로 확장
2 React Query 캐시 기반 서버 상태 관리 + 로컬 상태 분리
3 웹뷰 브리지 프로토콜 정리로 앱/웹 흐름을 하나의 경험으로 정렬
4 웹만으로도 탐색부터 결제까지 완성된 사용자 경험 구축

이중결제 이슈를 해소하기 위해 결제 흐름을 서버 중심으로 재설계한 사례

서버사이드 아키텍처 전환

문제 인식

결제 도중 이탈이나 중복 요청이 발생하면 이중결제와 결제 완료/주문 미기록 상태가 함께 생길 위험이 있었습니다. 주문 금액을 클라이언트 요청에 의존하는 구간이 남아 있어 금액 위변조 가능성이 있었고, 유료와 무료 주문의 처리 방식이 달라 주문/결제 데이터 정합성도 흔들릴 수 있었습니다.

분석

스케줄러로 사후 환불을 보정하는 방식만으로는 결제와 주문 사이 불일치를 근본적으로 해결하기 어렵다고 판단했습니다. 결제 성공 이후에 주문을 만드는 구조에서는 예외 상황이 생길 때마다 복구 비용이 커졌고, 무료 주문처럼 payment 레코드가 생기지 않는 경우에는 테이블 책임도 모호했습니다. 그래서 결제 로직을 API 한두 개 더 붙이는 수준이 아니라, 주문 생성과 결제 확정의 책임을 서버 기준으로 다시 설계해야 한다고 봤습니다.

실행

  • 주문 생성과 결제 확정의 책임을 서버가 갖는 구조로 결제 플로우를 다시 설계했습니다.
  • 주문 생성 단계에서 서버가 계산한 최종 금액과 클라이언트 요청 금액을 먼저 검증하고, 이후 Redis 세션의 가격 정보로 한 번 더 검증했습니다.
  • 주문 생성과 결제 확정을 분리하고, 사용자/도메인/가격을 기준으로 결제 세션을 검증하는 서버 기준 플로우로 바꿨습니다.
  • Bootpay confirm 타임아웃 시 PROCESSING 응답을 반환하고, 주문 상태 조회 API로 폴링할 수 있는 흐름을 추가했습니다.
  • WAITING_PAYMENT, COMPLETED_PAYMENT 같은 상태 전이를 서버 기준으로 재정리했습니다.
  • CARD 결제와 FREE 주문을 분기 처리해 무료 주문은 Bootpay 검증 없이 완료하고, 무료 주문에서도 정합성이 깨지지 않도록 로직을 수정했습니다.

결과

  • 반복적으로 발생하던 중복 결제 이슈를 해결했습니다.
  • 서버사이드 결제 구조 릴리즈 이후 동일 이슈가 재발하지 않았습니다.
  • 주문과 결제 데이터의 책임이 분명해졌고 예외 상황을 서버 기준으로 다룰 수 있게 됐습니다.

Before

1 Client → PG Provider: 결제창 호출
2 PG Provider: 카드 승인 완료
3 PG Provider → Client: onConfirm
4 Client → PG Provider: 즉시 결제 확정
5 PG Provider → Client: receiptId 전달
6 Client → Backend: 검증 및 DB 저장 요청
7 이 구간에서 실패하면 결제 완료 / 주문 미기록 상태가 남음

After

1 Client → Backend: 주문 생성 API 호출
2 Backend → Client: orderId / paymentId 반환
3 Client → PG Provider: 결제창 호출
4 PG Provider → Client: receiptId 전달
5 Client → Backend: 결제 확정 API 호출
6 Backend: 세션 검증, receipt 검증, 결제 상태 확정
7 Backend → Client: COMPLETED 또는 PROCESSING 응답
8 Client → Backend: 필요 시 상태 조회 API로 폴링
9 최종 주문 완료 상태를 서버 기준으로 확정

서비스 확장에 대비해 공통 코어 패키지와 초기화 구조를 표준화한 사례

공통 기반 패키지 설계

데이팅 기능이 별도 패키지로 분리되고 채팅 SDK까지 붙으면서, 새 기능이나 새 제품이 생길 때마다 인증, 로깅, 저장소, 네트워크 같은 기반 코드를 다시 만들어야 하는 상황이 됐습니다. 이 문제를 해결하기 위해 공통 인프라를 별도 패키지로 분리하고, 서비스별 구현은 그 위에 얹는 구조를 설계했습니다.

문제 인식

인증, 토큰, 로깅, 저장소, 인터셉터 같은 기반 기능이 여러 코드베이스에 흩어져 있으면 같은 성격의 보일러플레이트가 반복되고 초기화 충돌 위험도 함께 커집니다. 서비스가 늘어날수록 기능 개발보다 공통 인프라를 다시 맞추는 비용이 먼저 커지는 구조였습니다.

분석

단순히 파일을 한 폴더로 모으는 정도로는 다음 서비스 분리 때 같은 문제를 다시 겪게 된다고 판단했습니다. 공통화의 기준을 유틸 함수가 아니라 제품이 공통으로 기대하는 클라이언트 인프라로 잡아야, 새 패키지가 추가돼도 인증 방식, 로깅 규칙, 초기화 순서가 흔들리지 않는 구조를 만들 수 있었습니다. 그래서 코어는 공통 인프라를 책임지고, 앱과 패키지는 비즈니스 구현만 얹는 방향으로 계층을 나눴습니다.

실행

  • 새 제품이나 SDK가 붙어도 공통 기반을 그대로 재사용할 수 있도록 munto-core 패키지를 설계했습니다.
  • 저장소, 토큰, 로깅, 국제화, 네트워크, 채팅 SDK처럼 반복 구현되던 기능을 코어 계층으로 옮겼습니다.
  • CoreDependencyInjection을 중심으로 공통 인프라 초기화와 서비스별 초기화를 분리했습니다.
  • Logger, 결제, 본인인증은 인터페이스와 구현체를 분리해 서비스별로 교체 가능하도록 설계했습니다.
  • 문토, 데이팅, 채팅이 같은 사용자 인증 상태를 공유할 수 있도록 토큰 관리를 공통 TokenService로 통합하는 방향으로 설계했습니다.

결과

  • 새 패키지나 SDK를 붙일 때 인증, 로깅, 저장소, 네트워크 코드를 처음부터 다시 만드는 비용을 줄일 수 있는 기반을 만들었습니다.
  • 공통 기능 수정이 필요할 때 여러 코드베이스를 동시에 손보지 않고 코어 계층에서 일관되게 관리할 수 있는 구조를 갖췄습니다.
  • 서비스별 비즈니스 코드와 공통 인프라의 경계를 분리해, 제품 분리와 확장을 더 낮은 비용으로 진행할 수 있게 했습니다.

Before

1 Munto App과 Dating 서비스가 공통 기반 기능을 각각 별도 구현
2 Logger, i18n, Network Interceptor, Crashlytics 이벤트 스키마 중복 증가
3 토큰 관리, 로깅, DI 초기화, 빌드 설정(dev/prod) 방식이 서로 다른 구조로 굳어짐
4 새 서비스 추가 시 복제 비용과 충돌 위험 증가

After

Munto App
Dating Service
munto-core 공통 초기화·의존성 계층
TokenService
Logger + Extension Events
Storage / Environment / Build Config
i18n / Network / Crashlytics / Firebase / Sendbird

서로 다른 데이터 모델을 일관된 인터랙티브 차트 경험으로 통합한 사례

차트 시스템 설계

문제 인식

데이터 타입별로 화면을 따로 만들면 지표마다 계산 로직, 렌더링 방식, 인터랙션 처리가 분리되어 같은 앱 안에서도 차트 경험이 제각각이 될 위험이 컸습니다. 신규 지표가 추가될 때마다 화면을 다시 만드는 구조가 되면 개발 속도는 느려지고 유지보수 비용은 계속 증가할 수밖에 없었습니다.

분석

데이터 종류마다 화면을 따로 만드는 방식은 처음엔 빠를 수 있어도, 건강 데이터가 늘어날수록 차트 로직과 UX가 쉽게 분기될 수 있다고 봤습니다. 또 서버 DTO를 그대로 UI에 연결하면 데이터 타입이 늘어날 때마다 화면과 렌더링 로직이 함께 흔들릴 수 있었습니다. 그래서 DTO와 앱 모델을 분리하고, 공통 차트 모델과 더 낮은 렌더링 레이어를 먼저 설계하는 쪽이 재사용성과 UX 일관성 모두에 유리하다고 판단했습니다.

실행

  • 서버 DTO와 앱 모델 레이어를 분리하고, 팩토리 메소드 기반으로 데이터별 모델을 공통 차트 모델로 변환하는 구조를 설계했습니다.
  • 혈압의 min/max 값과 혈당·체중의 single value처럼 형태가 다른 데이터도 같은 렌더링 구조에서 처리할 수 있도록 설계했습니다.
  • CustomPainter와 GestureDetector 기반으로 차트 렌더링과 줌/스와이프 인터랙션을 직접 구현했습니다.

결과

  • 서로 다른 건강 데이터를 하나의 차트 시스템으로 일관되게 표현할 수 있게 됐습니다.
  • lerp 기반 애니메이션과 제스처 처리로 요구사항에 맞는 차트 UX를 구현했습니다.
  • 차트 로직의 재사용성과 확장성이 높아져 이후 건강 데이터 타입을 추가하기 쉬워졌습니다.

구현 흐름

1 혈압, 혈당, 체중 등 각기 다른 DTO 입력
2 DTO와 앱 모델 레이어 분리 + Mapper 규칙 정의
3 팩토리 메소드 기반 공통 차트 모델로 변환
4 CustomPainter 기반 렌더링
5 GestureDetector 기반 줌 / 스와이프 / 선택 인터랙션 적용
6 동일 구조 재사용으로 일관된 인터랙티브 차트 UX 구현
