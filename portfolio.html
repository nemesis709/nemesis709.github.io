앱 전환 중심 구조를 웹 완결형 사용자 경험으로 재구성한 사례

메인 화면 기준으로 기존 앱과 신규 웹(모바일/데스크톱) 비교

배경

모바일 서비스는 충분히 성장했지만, 웹은 소개 페이지 역할에 머물러 있었습니다. 로그인 상태나 회원 프로필, 결제 같은 실제 서비스 플로우가 웹에는 없었기 때문에 유입 이후 사용자가 머물 이유가 부족했습니다.

문제 인식

앱 소개용 링크 페이지로는 웹 유입을 실제 사용 전환으로 연결하기 어려웠습니다. 웹에서도 로그인, 상태관리, 결제까지 이어지는 완결된 경험이 필요했습니다. 기존 웹은 소개 페이지 중심이어서 실제 서비스 화면에서 필요한 회원 상태, 프로필, 인증 토큰 같은 상태를 다루는 구조가 없었습니다.

개선 방식

  • 앱 소개 중심이던 웹을 실제 서비스 진입점으로 바꾸고, 탐색부터 결제까지 이어지는 흐름을 웹 안에서 완결할 수 있도록 재구성했습니다.
  • 로그인 상태와 서비스 데이터를 안정적으로 다룰 수 있는 상태 관리 구조를 정리해 주요 화면 경험을 일관되게 맞췄습니다.
  • 웹에서 구현한 핵심 화면을 앱에서도 재사용할 수 있도록 연결 구조를 정리해 플랫폼별 기능 중복을 줄였습니다.

결과

  • 웹 MAU가 8만에서 12만으로 증가했고, 전체 MAU는 25만+ 규모로 확장됐습니다.
  • 앱과 웹의 사용자 흐름을 일관되게 맞춰 기능 확장 시 혼선을 줄였습니다.
  • 웹에서 구현한 VOD 경험을 앱 상세/플레이어 화면에서도 재사용할 수 있게 만들어 플랫폼별 기능 중복을 줄였습니다.
  • 웹뷰 캐시와 브리지 기반 통신을 통해 앱 안에서도 웹 기능을 자연스럽게 연결할 수 있는 구조를 만들었습니다.

Before

1 웹은 서비스 소개와 앱 다운로드 유도 중심의 랜딩 페이지
2 실제 탐색, 로그인, 결제는 모두 앱에서만 가능
3 웹 유입이 실제 사용 전환으로 이어지기 어려움
4 웹과 앱의 사용자 흐름이 분리되어 이용 경험이 자주 끊김

After

1 서비스 상세 페이지를 실제 서비스 진입점으로 확장
2 React Query 캐시 기반 서버 상태 관리 + 로컬 상태 분리
3 웹뷰 브리지 프로토콜 정리로 앱/웹 흐름을 하나의 경험으로 정렬
4 웹만으로도 탐색부터 결제까지 완성된 사용자 경험 구축

이중결제 이슈를 해소하기 위해 결제 흐름을 서버 중심으로 재설계한 사례

배경

기존 주문 흐름은 클라이언트 중심 결제 구조였고, 결제가 끝난 뒤 주문 데이터와 결제 데이터가 함께 생성되는 방식이었습니다. 결제 확정의 마지막 책임이 클라이언트에 남아 있는 구조였습니다.

문제 인식

결제 도중 이탈이나 중복 요청이 발생하면 이중결제와 결제 완료/주문 미기록 상태가 함께 생길 위험이 있었습니다. 주문 금액을 클라이언트 요청에 의존하는 구간이 남아 있어 금액 위변조 가능성이 있었고, 유료와 무료 주문의 처리 방식이 달라 주문/결제 데이터 정합성도 흔들릴 수 있었습니다.

개선 방식

  • 기존 클라이언트 중심 결제 구조에서 서버 중심 결제 구조로 전환하고, 주문 생성과 결제 확정을 분리해 결제 책임을 서버로 이관했습니다.
  • 주문 생성 단계에서 서버 계산 금액과 클라이언트 요청 금액을 먼저 대조하고, Redis 결제 세션으로 무결성 검증을 추가했습니다.
  • 서버에 WAITING_PAYMENT 상태를 도입해 결제 시도 단위의 주문번호를 먼저 고정하고, 재시도·재진입에도 동일 주문으로 처리되도록 멱등성을 보장했습니다.
  • Bootpay confirm 타임아웃 시 PROCESSING 응답과 상태 조회 폴링 흐름을 추가해, 이탈 후 재진입해도 같은 주문번호 기준으로 결제 상태를 이어서 확인할 수 있게 했습니다.
  • 유료/무료 주문을 분리 처리하되, 무료 주문에서도 결제 레코드 정합성이 유지되도록 동일한 데이터 규칙을 적용했습니다.

결과

  • 결제 진행 중 상태를 서버에서 관리하면서, 이탈 후 재진입 시 같은 주문번호로 상태를 이어갈 수 있는 구조를 만들었습니다.
  • 결제 확정 책임이 서버로 이동하면서 이중결제와 결제 완료 후 주문이 기록되지 않는 상태를 구조적으로 줄였습니다.
  • 유료/무료 주문 모두에서 주문/결제 데이터 정합성을 일관되게 유지할 수 있는 기준을 만들었습니다.
  • 타임아웃이나 지연 상황에서도 주문 상태를 추적할 수 있어 결제 운영 안정성과 대응 속도를 높였습니다.

Before

1 Client → PG Provider: 결제창 호출
2 PG Provider: 카드 승인 완료
3 PG Provider → Client: onConfirm
4 Client → PG Provider: 즉시 결제 확정
5 PG Provider → Client: receiptId 전달
6 Client → Backend: 검증 및 DB 저장 요청
7 이 구간에서 실패하면 결제 완료 / 주문 미기록 상태가 남음

After

1 Client → Backend: 주문 생성 API 호출
2 Backend → Client: orderId / paymentId 반환
3 Client → PG Provider: 결제창 호출
4 PG Provider → Client: receiptId 전달
5 Client → Backend: 결제 확정 API 호출
6 Backend: 세션 검증, receipt 검증, 결제 상태 확정
7 Backend → Client: COMPLETED 또는 PROCESSING 응답
8 Client → Backend: 필요 시 상태 조회 API로 폴링
9 최종 주문 완료 상태를 서버 기준으로 확정

서비스 확장에 대비해 공통 코어 패키지와 초기화 구조를 표준화한 사례

배경

문토 안의 부가 기능 수준이던 데이팅을 별도 서비스로 확장하면서, 로거, 국제화, 네트워크 같은 기반 모듈을 서비스별로 반복 구현하지 않고 공통화할 필요가 커졌습니다.

문제 인식

공통 기능이 서비스별로 분산된 상태에서는 같은 사용자 행동에도 서비스마다 인증, 로깅, 네트워크 동작이 달라질 위험이 컸고, Crashlytics 이벤트 수집 방식과 개발/프로덕션 빌드 설정이 서비스마다 달라, 장애 원인 추적과 릴리즈 검증 기준을 통일하기 어려웠습니다. 데이팅을 신규 서비스로 출시하는 시점에 구조를 정리하지 않으면 이후 서비스 확장 때마다 공통 모듈을 다시 구현해야 해 출시 속도와 품질이 함께 떨어질 수 있었습니다.

개선 방식

  • 앱 분리를 염두에 두고 공통 패키지 munto-core를 설계했습니다.
  • App, Device, Environment, Storage, Logger, Token, i18n, Network, Firebase, Sendbird를 공통 기반으로 분리했습니다.
  • 코어 초기화와 앱별 초기화를 분리해 서비스 간 초기화 충돌을 줄일 수 있는 구조로 정리했습니다.
  • 결제, 본인인증, 국제화처럼 서비스별로 달라질 수 있는 영역은 인터페이스와 구현체를 분리해 교체 가능하도록 구성했습니다.
  • 토큰 관리는 서비스별 보관 방식이 아니라 공통 토큰 서비스 기준으로 일원화해 인증 동작의 일관성을 확보했습니다.
  • 로거는 코어에서 로그 레벨과 출력 경로를 표준화하고, 레벨+아웃풋 조합에 따라 Fatal 로그를 Crashlytics로 전달하는 등 처리 경로를 분리해 운영 목적에 맞게 확장할 수 있도록 설계했습니다.

결과

  • 문토 앱과 데이팅 서비스가 공유하는 기반 기능의 책임을 한 곳으로 모을 수 있는 구조를 만들었습니다.
  • 공통 기능 수정 시 영향 범위를 줄이고 서비스별 코드 경계를 더 명확하게 만들었습니다.
  • 단순한 코드 이동이 아니라, 제품 경계와 초기화 계층을 기준으로 공통 코어를 설계하는 경험을 쌓았습니다.

Before

1 Munto App과 Dating 서비스가 공통 기반 기능을 각각 별도 구현
2 Logger, i18n, Network Interceptor, Crashlytics 이벤트 스키마 중복 증가
3 토큰 관리, 로깅, DI 초기화, 빌드 설정(dev/prod) 방식이 서로 다른 구조로 굳어짐
4 새 서비스 추가 시 복제 비용과 충돌 위험 증가

After

Dating Service
munto-core 공통 초기화·의존성 계층
TokenService

서로 다른 데이터 모델을 일관된 인터랙티브 차트 경험으로 통합한 사례

배경

사용자에게는 하나의 건강 기록 경험처럼 보여야 했지만, 실제 데이터는 타입마다 형식과 표현 범위가 달랐습니다. 상단 그래프는 줌인, 줌아웃, 스와이프, 날짜 선택까지 가능한 인터랙션을 가져야 했습니다.

문제 인식

데이터 타입별로 화면을 따로 만들면 지표마다 계산 로직, 렌더링 방식, 인터랙션 처리가 분리되어 같은 앱 안에서도 차트 경험이 제각각이 될 위험이 컸습니다. 신규 지표가 추가될 때마다 화면을 다시 만드는 구조가 되면 개발 속도는 느려지고 유지보수 비용은 계속 증가할 수밖에 없었습니다.

개선 방식

  • 차트 화면을 지표별 페이지가 아니라 공통 차트 엔진으로 설계하고, 데이터는 공통 모델로 변환해 주입하는 구조로 바꿨습니다.
  • 혈압처럼 범위값이 필요한 지표와 체중처럼 단일값 지표를 모두 같은 입력 규칙으로 수용할 수 있게 데이터 스펙을 정리했습니다.
  • 렌더링과 인터랙션 로직을 공통 레이어로 분리해 줌, 스와이프, 날짜 선택 경험이 지표와 무관하게 동일하게 동작하도록 맞췄습니다.
  • 새로운 지표가 들어와도 화면을 새로 만들지 않고, 변환 규칙만 추가하면 기존 차트 흐름에 바로 탑재할 수 있게 구성했습니다.

결과

  • 서로 다른 건강 데이터를 하나의 차트 시스템으로 일관되게 표현할 수 있게 됐습니다.
  • lerp 기반 애니메이션과 제스처 처리로 요구사항에 맞는 차트 UX를 구현했습니다.
  • 차트 로직의 재사용성과 확장성이 높아져 이후 건강 데이터 타입을 추가하기 쉬워졌습니다.
