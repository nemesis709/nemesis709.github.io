모바일 중심 서비스를 웹에서도 완결된 경험으로 확장한 경험

Web MAU 8만 → 12만 · 전체 MAU 25만+

기존 웹은 앱 소개 페이지에 가까웠고, 실제 서비스 탐색과 결제는 모두 앱으로 보내는 구조였습니다. 웹 유입이 실제 MAU 전환으로 잘 이어지지 않았고, 웹만으로 서비스를 이용하고 싶어하는 수요도 확인되고 있었습니다.

Context

모바일 서비스는 충분히 성장했지만, 웹은 소개 페이지 역할에 머물러 있었습니다. 로그인 상태나 회원 프로필, 결제 같은 실제 서비스 플로우가 웹에는 없었기 때문에 유입 이후 사용자가 머물 이유가 부족했습니다.

Problem

앱 소개용 링크 페이지로는 웹 유입을 실제 사용 전환으로 연결하기 어려웠습니다. 웹에서도 로그인, 상태관리, 결제까지 이어지는 완결된 경험이 필요했고, 앱과 웹의 라우팅 체계도 따로 놀고 있었습니다. 기존 웹은 소개 페이지 중심이어서 실제 서비스 화면에서 필요한 회원 상태, 프로필, 인증 토큰 같은 상태를 다루는 구조가 없었습니다.

Approach

  • 웹 플랫폼과 서비스 상세 페이지를 앱 디자인에 맞는 경험으로 고도화했습니다.
  • TanStack React Query를 중심으로 회원, 모임, VOD, 주문 도메인의 서버 상태와 캐시를 관리했습니다.
  • SSR 모바일 감지는 Context API로, 환경값과 브리지 상태는 훅 기반 로컬 상태로 분리해 역할별 책임을 나눴습니다.
  • 앱 딥링크를 go_router 기반으로 재정리해 웹과 앱이 호환되는 URI 구조를 만들었습니다.
  • VOD 상세와 플레이어는 웹에서 먼저 구현하고, 앱에서는 ContentWebViewController 기반 웹뷰로 같은 경로를 재사용할 수 있게 구성했습니다.
  • 브릿지는 요청/응답 메시지와 앱 이벤트 타입을 명시한 프로토콜로 정리하고, VOD 상세/플레이어 화면에서 토큰 요청, 좋아요 상태 조회, 화면 이동을 처리하도록 구성했습니다.
  • 웹만으로 로그인 이후 플로우와 결제를 이어갈 수 있게 실제 서비스 진입 구조를 다시 설계했습니다.

Result

  • 웹 MAU가 8만에서 12만으로 증가했고, 전체 MAU는 25만+ 규모로 확장됐습니다.
  • 웹과 앱의 URI 체계를 정리해 이후 기능 추가와 딥링크 유지보수 비용을 줄였습니다.
  • 웹에서 구현한 VOD 경험을 앱 상세/플레이어 화면에서도 재사용할 수 있게 만들어 플랫폼별 기능 중복을 줄였습니다.
  • 웹뷰 캐시와 브리지 기반 통신을 통해 앱 안에서도 웹 기능을 자연스럽게 연결할 수 있는 구조를 만들었습니다.

BEFORE

1 웹은 서비스 소개와 앱 다운로드 유도 중심의 랜딩 페이지
2 실제 탐색, 로그인, 결제는 모두 앱에서만 가능
3 웹 유입이 실제 사용 전환으로 이어지기 어려움
4 앱과 웹의 URI 체계가 분리되어 딥링크 유지보수도 어려움

AFTER

1 웹 플랫폼과 서비스 상세 페이지를 실제 서비스 진입점으로 확장
2 React Query 캐시 기반 서버 상태 관리 + 로컬 상태 분리
3 go_router 기반 URI 정리로 앱과 웹 경로 호환
4 웹과 앱이 같은 흐름을 공유하면서 Web MAU 8만 → 12만으로 성장

클라이언트 사이드 결제를 서버사이드 결제 구조로 전환한 경험

Munto App · Order v3

앱의 기존 결제 구조는 결제 성공 이후 주문과 결제 정보가 함께 생성되는 클라이언트 중심 플로우였습니다. 이 구조에서 이중 결제가 반복적으로 발생했고, 무료 주문에서는 결제 정보가 비어 정합성 문제도 함께 드러났습니다.

Context

기존 주문 API는 order/v2 기반이었고, 결제가 끝난 뒤 주문 데이터와 결제 데이터가 함께 생성되는 구조였습니다. 클라이언트가 결제 상태를 강하게 책임질수록 예외 상황을 다루기 어려웠고, 주문 가격도 클라이언트에서 신뢰하는 구간이 남아 있었습니다.

Problem

클라이언트 사이드 결제에서는 사용자가 결제 중 이탈하거나 중복 요청을 보내는 상황에서 상태를 안정적으로 보장하기 어려웠습니다. 특히 무료 주문은 payment 레코드가 생성되지 않아 order와 payment 테이블 사이 정합성도 흔들렸고, 서버 계산 금액과 클라이언트 요청 값을 분리해 검증할 필요가 있었습니다.

Approach

  • 기존 결제 구조를 서버사이드 결제 플로우로 전환하고 order/v3를 새로 설계했습니다.
  • 주문 생성 단계에서 서버가 계산한 최종 금액과 클라이언트 요청 금액을 먼저 검증하고, 이후 Redis 세션의 가격 정보로 한 번 더 검증했습니다.
  • 주문 생성과 결제 확정을 분리하고, 사용자/도메인/가격을 기준으로 결제 세션을 검증하는 서버 기준 플로우로 바꿨습니다.
  • Bootpay confirm 타임아웃 시 PROCESSING 응답을 반환하고, 주문 상태 조회 API로 폴링할 수 있는 흐름을 추가했습니다.
  • WAITING_PAYMENT, COMPLETED_PAYMENT 같은 상태 전이를 서버 기준으로 재정리했습니다.
  • CARD 결제와 FREE 주문을 분기 처리해 무료 주문은 Bootpay 검증 없이 완료하고, 무료 주문에서도 정합성이 깨지지 않도록 로직을 수정했습니다.

Result

  • 이중 결제 문제를 구조적으로 줄일 수 있는 기반을 만들었습니다.
  • 주문과 결제 데이터의 책임이 분명해졌고 예외 상황을 서버 기준으로 다룰 수 있게 됐습니다.
  • 핵심 비즈니스 플로우를 클라이언트 구현이 아니라 도메인 상태 기준으로 설계하는 경험을 했습니다.

AS-IS

1 Client → PG Provider: 결제창 호출
2 PG Provider: 카드 승인 완료
3 PG Provider → Client: onConfirm
4 Client → PG Provider: 즉시 결제 확정
5 PG Provider → Client: receiptId 전달
6 Client → Backend: 검증 및 DB 저장 요청
7 이 구간에서 실패하면 결제 완료 / 주문 미기록 상태가 남음

결제 확정 시점이 클라이언트에 있어 네트워크 오류나 앱 종료가 발생하면 결제는 완료됐지만 주문이 기록되지 않는 구간이 남아 있었습니다.

TO-BE

1 Client → Backend: 주문 생성 API 호출
2 Backend → Client: orderId / paymentId 반환
3 Client → PG Provider: 결제창 호출
4 PG Provider → Client: receiptId 전달
5 Client → Backend: 결제 확정 API 호출
6 Backend: 세션 검증, receipt 검증, 상태 전이 처리
7 Backend → Client: COMPLETED 또는 PROCESSING 응답
8 Client → Backend: 필요 시 상태 조회 API로 폴링
9 최종 주문 완료 상태를 서버 기준으로 확정

주문 생성과 결제 확정을 분리하고, 서버가 상태 전이와 정합성을 책임지도록 바꿔 예외 상황에서도 결제 상태를 추적할 수 있게 했습니다.

서비스 분리를 염두에 두고 앱 공통 기반 패키지를 설계한 경험

Shared Client Infrastructure

문토 앱, 데이팅 패키지, 채팅 SDK까지 공통 기능을 공유하는 범위가 넓어지면서 코어 패키지 분리를 다시 검토하게 됐습니다. 단순한 모듈 정리가 아니라, 두 번째 패키지 분리 시점에 맞춰 공통 의존성과 초기화 구조를 다시 정의한 작업이었습니다.

Context

초기에 공통 코어 분리를 검토했을 때는 아직 서비스 수가 적어 보류했습니다. 하지만 채팅 SDK가 추가되면서 SharedPreference, Logger, i18n, Network Interceptors, DI 초기화 같은 영역의 중복이 무시하기 어려운 수준이 됐습니다.

Problem

공통 기능이 앱, 패키지, SDK에 분산되면 중복 코드가 늘어나는 것뿐 아니라 토큰 관리, 로깅 방식, DI 초기화 순서까지 서로 다르게 굳어질 수 있었습니다. 서비스가 하나 더 늘어날수록 수정 비용이 기하급수적으로 커지는 구조였습니다.

Approach

  • 앱 분리를 염두에 두고 공통 패키지 munto-core를 설계했습니다.
  • App, Device, Environment, Storage, Logger, Token, i18n, Network, Firebase, Sendbird를 공통 기반으로 분리했습니다.
  • CoreDependencyInjection을 중심으로 코어 초기화와 앱별 초기화를 분리해 DI 충돌을 줄이는 구조를 제안하고 구현했습니다.
  • Logger 출력, 결제, 본인인증, 국제화처럼 서비스별로 바뀔 수 있는 영역은 인터페이스와 구현체를 분리해 교체 가능하도록 구성했습니다.
  • 토큰은 각 서비스가 따로 들고 있는 방식 대신 공통 TokenService 기준으로 통합하는 방향을 설계했습니다.
  • 로거는 코어에서는 기본 기능만 제공하고, 각 패키지에서 extension으로 비즈니스 이벤트를 확장하는 구조를 설계했습니다.

Result

  • 문토 앱, 데이팅 패키지, 채팅 SDK가 공유하는 기반 기능의 책임을 한 곳으로 모을 수 있는 구조를 만들었습니다.
  • 공통 기능 수정 시 영향 범위를 줄이고 서비스별 코드 경계를 더 명확하게 만들었습니다.
  • 단순한 코드 이동이 아니라, 제품 경계와 초기화 계층을 기준으로 공통 코어를 설계하는 경험을 쌓았습니다.

Before

1 Munto App, Dating Package, Chat SDK가 각자 공통 기능을 별도 구현
2 SharedPreference, Logger, i18n, Network Interceptors 중복 증가
3 토큰 관리, 로깅, DI 초기화 방식이 서로 다른 구조로 굳어짐
4 새 서비스 추가 시 복제 비용과 충돌 위험 증가

After

Munto App
Dating Package
Chat SDK
CoreDependencyInjection + munto-core
TokenService
Logger + Extension Events
Storage / Environment
i18n / Network / Firebase / Sendbird

다양한 건강 데이터를 하나의 인터랙티브 차트 시스템으로 추상화한 경험

Healthcare App · Flutter

혈압, 혈당, 키, 몸무게, 생리주기처럼 성격이 다른 건강 데이터를 기록하는 기능이 필요했습니다. 차트는 공통 UI처럼 보여야 했지만, 서버는 각 데이터별 DTO를 따로 내려주고 있었고 UX 요구사항도 까다로웠습니다.

Context

사용자에게는 하나의 건강 기록 경험처럼 보여야 했지만, 실제 데이터는 타입마다 형식과 표현 범위가 달랐습니다. 상단 그래프는 줌인, 줌아웃, 스와이프, 날짜 선택까지 가능한 인터랙션을 가져야 했습니다.

Problem

데이터별 화면을 각각 구현하면 차트 로직이 중복되고 UX 일관성도 깨질 수 있었습니다. 위젯 수준에서만 접근하면 요구사항에 맞는 렌더링과 애니메이션을 구현하기도 어려웠습니다.

Approach

  • 서버 DTO와 앱 모델 레이어를 분리하고, 팩토리 메소드 기반으로 데이터별 모델을 공통 차트 모델로 변환하는 구조를 설계했습니다.
  • 혈압은 min/max, 혈당과 체중은 single value처럼 데이터 형태가 다른 경우도 같은 렌더링 구조에 태울 수 있게 만들었습니다.
  • CustomPainter와 GestureDetector 기반으로 차트 렌더링과 줌/스와이프 인터랙션을 직접 구현했습니다.

Result

  • 서로 다른 건강 데이터를 하나의 차트 시스템으로 일관되게 표현할 수 있게 됐습니다.
  • lerp 기반 애니메이션과 제스처 처리로 요구사항에 맞는 차트 UX를 구현했습니다.
  • 차트 로직의 재사용성과 확장성이 높아져 이후 건강 데이터 타입을 추가하기 쉬워졌습니다.

Data Mapping

1 혈압, 혈당, 체중 등 각기 다른 DTO 입력
2 DTO와 앱 모델 레이어 분리
3 팩토리 메소드 기반 공통 차트 모델로 변환
4 같은 차트 렌더링 구조에서 재사용

4 요구사항에 맞는 인터랙티브 차트 UX 구현
